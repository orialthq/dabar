# 다바르 M5 — 로컬 AI 전환 기획서

> 목표: API 키 없이, 서버 없이, 모든 사용자의 브라우저에서 동작하는 AI 레이어.

## 0. 배경과 결정

M4는 BYOK(Anthropic API)로 AI를 얹었다. 그러나 타겟(일반 교인)에게 API 키는 구조적 장벽이며, 운영 방향도 API 비의존으로 확정되었다. 이에 AI 레이어를 로컬(브라우저 내) 실행으로 전환한다.

**핵심 아키텍처 결정: 말씀 추천은 생성이 아니라 검색이다.**
- "일상 기록 → 닿는 구절"은 의미 유사도 문제. 챗 LLM 불필요.
- 31,102절을 빌드 타임에 임베딩 → 양자화하여 앱에 동봉 → 브라우저에서 기록을 임베딩 → 코사인 유사도 top-k.
- 효과: 전 사용자 동작(키·서버 불필요), 할루시네이션 원천 불가능(검색이므로), 1회 다운로드 후 오프라인 동작, 비용 0.

**챗(묻다, 묵상 질문 생성)은 선택적 레이어.**
- WebLLM(WebGPU) 기반 소형 모델. 미지원 환경에서는 묻다만 비활성화하고 추천(M5a)은 항상 동작.
- 소형 모델의 성경 참조 암기는 신뢰 불가 → 참조는 임베딩 검색이 공급하고, LLM은 그 위에서 말만 얹는 구조로 역전한다 (아래 M5b).

## 1. 마일스톤 분할

### M5a — 임베딩 기반 말씀 추천 (코어)
1. **임베딩 파이프라인** (`scripts/embed-bible.mjs`, Node)
   - 모델: `intfloat/multilingual-e5-small` (MIT, 384차원). @huggingface/transformers(Node)로 실행.
   - 입력 형식: E5 규약 준수 — 절 텍스트는 `passage: {책이름} {장}:{절} {본문}`, 쿼리는 `query: {기록}`. **프리픽스 누락 시 성능 급락 주의.**
   - 짧은 절 보완 실험: 절 단독 임베딩 vs 앞뒤 1절 문맥 포함 임베딩 A/B 비교 후 결정 (평가 방법은 §3).
   - 출력: `public/embeddings/verses-{n}.bin` — int8 양자화(벡터별 scale, float16) + `meta.json`(절 인덱스 ↔ bookId/장/절 매핑, 샤드 정보). 예상 총량 ~13MB, 4샤드 분할.
2. **런타임 검색** (`src/lib/semantic.ts`)
   - @huggingface/transformers(브라우저)로 e5-small 로드. WebGPU 가용 시 WebGPU, 아니면 WASM. 모델은 브라우저 캐시(Cache API)에 저장 — 최초 1회 ~100MB 안내 UI 필수(진행률 표시, Wi-Fi 권장 문구).
   - 임베딩 샤드는 fetch + ArrayBuffer로 로드, 메모리 상 코사인 유사도 전수 계산(31k×384 int8 → 수십 ms 수준, 워커 스레드 권장).
   - top-k(기본 8) → 같은 장 중복 억제 → 상위 4개 반환.
3. **UI 교체**: Editor의 VerseSuggest를 임베딩 검색 기반으로 교체. "이 하루에 닿는 말씀 찾기" UX 유지. why 문구는 M5a에서는 생략(또는 유사도 기반 간단 라벨), 묵상 질문은 M5b로 이관.
4. **폴백**: 모델 로드 실패/거부 시 기존 키워드 검색(VersePicker)만 노출. 기능 전체가 죽지 않게.

### M5b — WebLLM 챗 레이어 (선택적)
1. **엔진 추상화** (`src/lib/engine.ts`): `generate(system, messages) → text` 인터페이스 하나로 통일. 구현체 3종:
   - `webllm`: @mlc-ai/web-llm, 기본 모델 **Qwen3-4B**(Apache 2.0, MLC 프리빌드 존재 여부 우선 확인; 없으면 Qwen2.5-3B-Instruct). 한국어 특화 후보(Kanana nano 계열)는 라이선스 및 MLC 변환 가능성 검증 후 교체 검토.
   - `ollama`: http://localhost:11434 (개발·파워유저용, 모델명 설정 가능)
   - `anthropic`: 기존 M4 BYOK 코드 이전 (옵션으로 존치)
   - 설정 화면에서 엔진 선택. 기본값 webllm.
2. **역전된 RAG**: LLM에게 참조를 묻지 않는다. ① 사용자 입력을 임베딩 검색 → 관련 절 top-k를 **본문 포함** 프롬프트에 공급 ② LLM은 공급된 절 안에서만 근거를 골라 답변 ③ 답변 내 구절 인용은 여전히 금지, 참조 표기만 → 앱이 로컬 DB로 렌더링(기존 resolveRefs 재사용).
3. **묻다 / 묵상 질문**: 위 구조로 재구현. WebGPU 미지원 브라우저에서는 묻다 진입 시 안내 + 비활성.

## 2. 불변 조건 (변경 없음)
- 구절 본문은 항상 로컬 성경 DB에서만 렌더링. LLM 생성/복사 본문 표시 금지.
- 개역한글 외 역본 금지. 본문 무변형.
- 서버 금지. 저장은 localStorage. 외부 전송은 사용자가 명시 선택한 엔진 호출뿐.
- 교단 중립, 정답 제시가 아닌 상고 유도 (기존 프롬프트 원칙 유지).

## 3. 검증 기준
- 파이프라인: 임베딩 절 수 = 31,102, meta 매핑 전수 일치, 샤드 로드 후 무작위 100절 재계산 대조.
- 추천 품질 골든셋: 일상 기록 시나리오 10건(위로/감사/불안/갈등/결정/피로 등) × 기대 구절 후보를 수동 작성 → top-4 적중률 기록. e5-small 결과가 기대 이하면 bge-m3 계열(용량 大) 비교 후 트레이드오프 결정.
- 성능 예산: 최초 모델 다운로드 외에, 검색 1회 응답 < 1.5s (중급 노트북 WASM 기준).
- 회귀: M1~M4 기능(리더/검색/새김/내보내기) 전부 기존대로 동작.

## 4. 리스크
- 모델 최초 다운로드(~100MB) 이탈 → 다운로드 전에도 앱 가치가 성립하도록(리더+키워드검색+수동 새김), 추천 기능은 "준비하기" 버튼으로 opt-in.
- iOS Safari WebGPU/메모리 제약 → WASM 경로 필수 테스트.
- Qwen3 소형 모델의 한국어 존댓말 톤 → 시스템 프롬프트로 보정, 골든셋으로 확인.
